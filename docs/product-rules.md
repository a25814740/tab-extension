# Product Rules

## 核心資訊架構
本產品沿用以下層級命名：

- 組織
- 空間
- 集合
- 頁籤

對應邏輯：
- 組織：最高層級容器
- 空間：組織底下的工作領域
- 集合：空間底下的 tab 群組
- 頁籤：單一網址 / 單一瀏覽頁

---

## 產品定位
這不是單純的 Toby clone，也不是社群通知中心。

本產品定位為：
- 新分頁工作台
- 分頁管理工具
- 工作場景切換器
- Dock 式快捷啟動列
- 常用入口與暫放區

---

## Dock 定義
Dock 的定位不是通知欄，而是高頻入口與快捷操作層。

Dock 主要用途：
- 核心功能入口
- 常用網站 / 工具入口
- 最近使用入口
- 暫放區
- 設定入口

Dock 不主打：
- 抓社群官方未讀
- 跨平台私訊整合
- 通知聚合中心

---

## 左側 Sidebar 規則
左側代表「結構導覽」，不是內容區。

左側負責：
- 組織切換
- 空間切換
- 空間操作
- 帳號 / 設定入口

帳號與設定主入口放左下角。

---

## 中間主區規則
中間主區代表「目前空間下的集合內容」。

中間主區負責：
- 顯示當前空間的集合
- 切換視圖
- 排序
- 搜尋
- 新增集合
- 管理集合

中間主區不需要顯示多餘標題「集合」字樣，breadcrumb 即可。

---

## 右側 Sidebar 規則
右側代表「目前瀏覽器開啟分頁狀態」。

右側負責：
- 顯示目前開啟的 tab
- 多選 tab
- 批次加入集合
- 批次加入 Dock
- 批次移到空間

右側不是收藏區，而是工作中的暫態分頁區。

---

## 新增規則
新增入口要分層：

- 左側空間區的新增：新增空間
- 中間集合區的新增：新增集合
- 組織區的新增：新增組織

不可混用。

---

## 編輯 / 刪除規則
- 刪除組織放在編輯組織內，不必獨立顯示在組織主區
- 組織主區只保留：
  - 新增組織
  - 編輯組織
  - 邀請好友

---

## 邀請好友規則
邀請好友有分層概念：

- 組織層邀請：邀請進整個組織
- 空間層邀請：邀請進某個空間
- 集合層邀請：分享某個集合

另外帳號區也可有一個全域型的邀請入口，用於產品推薦 / 邀請註冊。

---

## 設計原則
- 優先保留目前 UI 架構
- 不要任意重新設計整體風格
- 優先修正語法、版面、互動、結構問題
- 優先讓畫面可執行、可維護
- 優先遵守既有規格，不自行擴張需求

---

## 技術原則
- 使用 Tailwind CSS
- 元件應可重構，但不要破壞現有 UI 邏輯
- 若需拆檔，請保持專案可執行
- 若有 lint / type / build 問題，需一併修正
