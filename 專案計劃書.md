# Chrome 分頁管理擴充套件專案規劃書（Codex 接手版）

> 本文件用途：提供給 Codex / AI 開發代理 / 新加入工程師，讓其能直接依照本規劃書繼續完成專案。
>
> 本版本已依照最新商業決策重寫：**取消永久免費版，改為 60 天試用；試用後主推年費 NT$99 的個人版；低價版資料盡量儲存在使用者自己的 Google Drive；高階功能再使用自家雲端。**

---

## 1. 專案摘要

本專案是一個 Manifest V3 的 Chrome 擴充套件，核心目標是提供以工作流為核心的分頁整理系統，支援：

- 儲存目前分頁 / 目前視窗 / 分頁群組
- 以 Space / Folder / Collection 結構整理分頁
- 拖拉排序、搜尋、去重、批次開啟
- 側邊欄快速操作
- 試用期內可啟用雲端同步
- 年費個人版以「使用者自有 Google Drive 儲存」為主，降低平台成本
- 未來可擴充成自家雲端同步、分享協作、AI 接口、團隊版與企業版

本專案採 **local-first** 設計：

- 本地操作永遠優先，不阻塞 UI
- 優先儲存在 `chrome.storage.local`
- 需要同步時，再寫入使用者自己的 Google Drive `appDataFolder`
- 商業與會員資料則儲存在自家後端（Supabase）

## 2. 最終產品策略（本版定案）

### 2.1 收費與方案定位

本專案不提供永久免費版。

改採以下模式：

1. **60 天完整試用**
   - 讓使用者完整體驗核心流程
   - 試用期間可使用同步功能
   - 試用期間資料可同步到使用者自己的 Google Drive

2. **試用到期後的個人版（主推）**
   - **年費 NT$99**
   - 提供個人使用的整理、儲存、搜尋、同步
   - 同步資料儲存在使用者自己的 Google Drive `appDataFolder`
   - 不提供團隊協作、公開分享頁、自家重度雲端功能

3. **進階版 / Pro（月費）**
   - 預留未來方案，例如 NT$99/月 或 NT$149/月
   - 才開放自家雲端同步、快照歷史、協作、分享、進階搜尋、多人空間等高成本功能

4. **AI 功能**
   - 採 BYO API Key
   - 專案只提供 Provider 接口與 UX
   - 專案方不負擔 AI Token 成本

### 2.2 商業邏輯

此策略的目標是：

- 降低早期固定成本與邊際成本
- 不養永久免費用戶
- 讓低價用戶不要消耗自家儲存與同步資源
- 讓高成本功能只開給高價方案
- 保留之後升級到 Workona 類型產品的空間

## 4. 成本控制策略（關鍵決策）

### 4.1 為什麼採 Google Drive `appDataFolder`

低價版若使用自家雲端來保存大量 Tab/Collection/快照資料，會導致：

- DB / Storage 成本上升
- 出站流量成本上升
- 同步 API 成本上升
- 客服與資料修復負擔上升

因此本版決定：

- **使用者內容資料**：存使用者自己的 Google Drive `appDataFolder`
- **商業資料 / 會員資料 / 試用與付費狀態**：存自家 Supabase

### 4.2 哪些資料存在使用者自己的 Google

以下內容儲存在 `appDataFolder`：

- spaces
- folders
- collections
- tabItems
- sort order
- notes
- settings snapshot
- dedupe cache（可選）
- 最近一次完整 state snapshot

建議儲存格式：

- 單一主檔 `workspace_state.json`
- 或分拆為：
  - `meta.json`
  - `spaces.json`
  - `collections.json`
  - `tabs.json`

初版建議先採 **單一主檔 + version 欄位**，減少同步複雜度。

### 4.3 哪些資料存在自家 Supabase

Supabase 只保存：

- user account 對應資料
- 試用開始日 / 到期日
- 付費狀態
- plan type
- payment provider reference
- invitation code / referral 統計
- feature flags
- optional telemetry（匿名）
- audit logs（必要時）

**不要**在 Personal 年費版把主要 Tab/Collection 內容寫入 Supabase。

## 6. 技術架構

### 6.1 前端 / 擴充套件架構

建議 monorepo：

- `apps/newtab`：New Tab 主介面
- `apps/sidepanel`：側邊欄介面
- `apps/options`：設定頁 / 會員與同步管理
- `packages/core`：資料模型、狀態管理、同步邏輯、schema
- `packages/chrome-adapters`：Chrome API 封裝
- `packages/google-drive-sync`：Google OAuth、Drive API、序列化與同步
- `packages/api-client`：自家後端 API client（Supabase / payment / membership）
- `packages/ai`：BYO AI provider 抽象（先留 interface）
- `extension/`：manifest 與打包輸出

### 6.2 狀態管理策略

採 local-first：

- UI 操作 => 更新本地 store
- 寫入 `chrome.storage.local`
- 若使用者已授權 Google 且會員有效 => 排程同步至 Google Drive
- 若同步失敗 => 本地標記 dirty / retry

建議 store 結構：

```ts
AppState {
  user: UserState
  membership: MembershipState
  sync: SyncState
  workspace: WorkspaceState
  ui: UIState
}
```

### 6.3 同步架構

同步方向：

- 本地 -> Google Drive
- Google Drive -> 本地還原

初版同步原則：

- 單人使用為主
- 同一帳號多裝置可同步
- 衝突策略先採：**latest updatedAt wins**
- 以完整 state snapshot 為主
- 可選擇加上簡化 diff/pending ops，但不要讓首發過度複雜

### 6.4 為什麼初版不優先做複雜 pending ops 雲端同步

因為本版主要同步目的不是多人共編，而是：

- 備份
- 跨裝置恢復
- 讓低價版有「雲端同步」體驗

因此初版可接受：

- 本地狀態整包序列化
- 上傳到 `appDataFolder`
- 下載時檢查版本號與 updatedAt

這會比自建複雜 op-log / merge engine 更快達成穩定上線。

## 7. Google 整合設計

### 7.1 Google OAuth

Chrome Extension 使用：

- `chrome.identity.getAuthToken()` 取得 Google access token
- 以 manifest 中的 `oauth2` 設定 client ID 與 scopes

需要 scope：

- `https://www.googleapis.com/auth/drive.appdata`

注意事項：

- 不要在安裝後首次啟動就強制跳授權
- 應在使用者主動點擊「啟用同步」時觸發授權
- UI 要明確說明：資料將儲存在使用者自己的 Google Drive 應用程式隱藏資料夾

### 7.2 Google Drive 儲存設計

使用：

- `appDataFolder`

建議檔案：

- `workspace_state_v1.json`

內容欄位建議：

```json
{
  "version": 1,
  "updatedAt": "2026-03-21T10:00:00.000Z",
  "deviceId": "xxx",
  "workspace": { ... }
}
```

流程：

1. 啟用同步時先查找 `appDataFolder` 中是否已有主檔
2. 若沒有則建立新檔
3. 若有則讀取並比對 `updatedAt`
4. 視情況選擇：
   - 以雲端覆蓋本地
   - 以本地覆蓋雲端
   - 顯示簡易衝突提示

### 7.3 同步頻率

初版建議：

- 手動 Sync Now
- 加上 debounce 自動同步（例如本地變更後 10~30 秒）
- 啟動時檢查遠端版本

不要一開始就做高頻 background sync，以免增加 API 呼叫與錯誤複雜度。

## 8. 會員與付款架構

### 8.1 會員狀態定義

```ts
MembershipStatus =
  | 'trial_active'
  | 'trial_expired'
  | 'paid_active'
  | 'paid_expired'
  | 'cancelled'
```

### 8.2 會員判斷規則

- 新註冊使用者：建立 `trial_started_at`
- 預設試用期：60 天
- 若當前日期 <= `trial_started_at + 60 days` => `trial_active`
- 若試用過期且無有效付款 => `trial_expired`
- 若有有效年費付款且未到期 => `paid_active`

### 8.3 試用到期後的行為

- 保留本地資料
- 停用雲端同步按鈕
- 設定頁顯示升級 CTA
- 不刪除使用者本機資料
- 允許匯出 JSON

### 8.4 統一金流

本版需求：

- 至少支援年費 NT$99 購買
- 付款完成後更新 Supabase 的會員狀態
- 要能防止前端偽造付款成功

建議做法：

1. 前端向自家 API 請求建立訂單
2. API 呼叫統一金流
3. 使用 webhook / callback 更新付款狀態
4. 後端寫入會員啟用與到期日
5. 前端重新抓取 membership status

### 8.5 金流邏輯重點

- 年費 99 優先，不做複雜月費與點數制
- 先只開刷卡
- 付款邏輯都在後端驗證，不信任前端
- 每一筆付款必須有 provider transaction id 可追蹤

## 12. 非功能需求

### 12.1 穩定性

- 本地資料不能因同步失敗而遺失
- 任一同步錯誤不得阻塞使用者繼續整理分頁
- 所有 destructive actions 需二次確認或可復原

### 12.2 效能

- 200~1000 tabs 仍需基本可用
- 搜尋需有 debounce
- 大量 collections 載入需避免整頁卡死

### 12.3 安全性

- Google token 不永久明文存放
- 付款狀態只信任後端
- API key（未來 BYO AI）需加密或至少以瀏覽器安全儲存策略處理
- 不要求多餘 Google 權限

### 12.4 可維護性

- schema versioning
- migration strategy
- 模組化 packages
- 至少有基本單元測試

## 13. 建議技術細節

### 13.1 狀態同步策略建議

初版建議用：

- `chrome.storage.local` 作為本地真實來源
- Google Drive 作為遠端備份與跨裝置恢復來源
- 採 snapshot-based sync

不要一開始就：

- 實作複雜 CRDT
- 實作多人同改 merge engine
- 實作複雜 server-side operation logs

### 13.2 JSON schema versioning

建議所有匯出與雲端檔案都包含：

```json
{
  "schemaVersion": 1,
  "appVersion": "x.y.z",
  "updatedAt": "..."
}
```

便於未來升級時做 migration。

### 13.3 錯誤處理

同步錯誤需分級：

- OAuth failed
- Token expired
- Drive read failed
- Drive write failed
- Membership invalid
- Payment pending
- Local schema invalid

UI 需能顯示最後錯誤資訊。

## 15. 驗收標準（Done Definition）

首發版本視為完成的條件：

1. 使用者可安裝擴充套件
2. 可在新分頁頁面與側邊欄操作
3. 可保存目前分頁 / 視窗
4. 可建立 Space / Folder / Collection
5. 可拖拉整理與搜尋
6. 可去重與一鍵開啟 Collection
7. 可登入 Google
8. 可同步到自己的 Google Drive
9. 試用 60 天邏輯正確
10. 可購買 NT$99 年費
11. 付款後會員狀態可正確更新
12. 試用過期後會停用同步，但不破壞本地資料
13. 匯出/匯入可正常運作
14. build / lint / basic test 可通過

---

## 16. 給 Codex 的明確開發限制

1. **不要把個人版主要內容資料寫進 Supabase**
2. **不要先做團隊協作與分享**
3. **不要先做複雜即時同步**
4. **不要導入高成本 AI 代管流程**
5. **優先確保本地可用與 Google Drive 同步穩定**
6. **付款流程必須由後端驗證，不信任前端成功訊號**
7. **所有權限與 scope 需最小化**

## 18. 實作提示（供 Codex 直接使用）

如果 Codex 要直接開始實作，優先事項如下：

1. 先完成本地資料模型與 UI
2. 再完成 Google Drive `appDataFolder` 同步
3. 再做 Supabase membership 與 trial
4. 最後再接統一金流

必要時可先以 mock payment provider 完成會員流程，再替換為正式 PAYUNi。

## 19. 一句話總結

這個專案的首發版本，不是要先做成高成本的 Workona，而是要先做成：

**一個本地優先、低成本、可試用、可年費續用、並將個人資料儲存在使用者自己 Google Drive 的 Chrome 分頁管理工具。**
